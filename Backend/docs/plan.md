## 1. 總體目標

後端要從目前的“骨架版”進化為下面這幾件事的整體系統：

- 支援用戶註冊與登入
- 支援 chunk 的查詢與學習流程
- 支援學習進度與每日目標
- 支援設置偏好
- 支援 AI 批量生成內容，並把內容存入共享池
- 最終能與前端順利串接

---

## 2. 建議的實施路線

我建議你把後端分成 5 個階段來做，而不是一口氣做完，這樣更容易驗證，也更容易出問題時定位。

---

## 3. 階段一：先把基礎打實

這一步的目標是讓後端從“只有框架”變成“能真正存資料、能真正處理請求”的版本。

### 你要完成的內容
1. 接上 PostgreSQL
2. 用 Prisma 生成資料表
3. 實現真實的註冊 / 登入邏輯
4. 用 bcrypt 做密碼加密
5. 用 JWT 做身份驗證

### 你要新增的能力
- 用戶可以註冊
- 用戶可以登入
- 系統能識別是誰在請求

### 需要注意的問題
- 資料庫連接失敗
- 環境變量配置錯誤
- JWT 秘鑰配置不一致
- 密碼加密與驗證邏輯錯誤

### 建議的驗證方式
- 手動建立一個測試帳號
- 用註冊接口創建帳號
- 用登入接口拿到 token
- 用 /auth/me 驗證身份

---

## 4. 階段二：把內容系統從 mock 改成真實資料庫

這一步是讓前端真正能拿到內容。

### 你要完成的內容
1. 把目前寫死的 chunk 資料替換成資料庫資料
2. 實現 chunk 列表接口
3. 實現隨機 chunk 接口
4. 實現例句查詢
5. 實現 quiz 問題查詢

### 你要新增的能力
- 前端能拿到 chunk
- 前端能看到例句
- 前端能看到填空題與選擇題

### 需要注意的問題
- 前端需要的資料格式要和後端返回格式一致
- chunk 的欄位命名要和前端約定好
- 同一個 chunk 的例句和題目要能關聯
- 查詢過程中可能出現查不到資料的情況

### 建議的驗證方式
- 用 /chunks 接口查詢資料
- 用 /chunks/random 拿一條資料
- 確認返回結構包含 phrase、translation、examples、options 等必要字段

---

## 5. 階段三：完善學習與進度系統

這一步是讓 app 不只“顯示內容”，而是“能真正記錄學習”.

### 你要完成的內容
1. 用戶答題後，記錄答案
2. 判斷答對 / 答錯
3. 更新 mastered / needsReview 狀態
4. 記錄每日完成數量
5. 計算 streak 和每日目標

### 你要新增的能力
- 用戶每次回答後，進度會被記錄
- 系統知道哪些 chunk 已掌握
- 系統知道哪些 chunk 還需要複習

### 需要注意的問題
- 答題結果要正確落庫
- 同一個 chunk 的多次回答要合理累積
- 日志和 streak 的計算要避免邊界錯誤
- 同一天重複提交可能造成數據重複

### 建議的驗證方式
- 模擬一個用戶答題多次
- 驗證 progress 表是否更新
- 驗證 daily progress 是否正確累加
- 驗證 mastered 與 review 狀態是否符合邏輯

---

## 6. 階段四：接入 AI 批量生成內容

這是你最重要的產品特點，應該作為後端的核心能力之一。

### 你要完成的內容
1. 建立 generation job 任務
2. 定時觸發內容生成
3. 調用 AI API 批量生成：
   - chunks
   - 例句
   - quiz 題目
4. 把生成結果寫入內容池
5. 加入內容狀態流轉

### 建議的狀態流
- draft
- generating
- pending_review
- published
- retired

### 你要新增的能力
- 不是每個用戶請求時都生成內容
- 而是定時批量生成，存入內容池
- 用戶學習時直接從池子中取資料

### 需要注意的問題
- AI 成本過高
- 生成內容品質不穩
- 生成內容重複
- AI 服務超時或掛掉
- 生成任務失敗時怎麼處理

### 建議的解決方案
- 用 queue / background job 處理
- 批量生成，而不是逐個請求
- 生成後先做檢查，再發佈
- 生成失敗時記錄 job 狀態與錯誤原因

---

## 7. 階段五：與前端完成真正連接

這一步是把前端 UI 和後端 API 真正接上。

### 你要完成的內容
1. 前端登入接口接上
2. 前端 chunk 資料從後端拉取
3. 前端練習結果提交到後端
4. 前端設定存到後端
5. 確保 CORS、headers、token 傳遞正確

### 需要注意的問題
- CORS 配置
- 前端請求頭缺少 authorization
- API 返回格式和前端預期不一致
- 前端使用的欄位名與後端不一致

### 建議的驗證方式
- 前端可以註冊 / 登入
- 前端可以打開 Home 頁拿到 chunk
- 點選答案後進度會更新
- 設定頁的選項能被保存

---

## 8. 測試計劃

測試不能只靠“看起來有反應”，要分成幾層來做。

### 8.1 單元測試
測服務層邏輯，例如：
- 密碼加密
- JWT 生成
- 智能判斷是否需要 review
- 每日目標計算

### 8.2 整合測試
測 API 與資料庫是否正常工作，例如：
- 註冊流程
- 登入流程
- 取得 chunk
- 提交答案
- 更新設定

### 8.3 手動測試
你自己至少要測這些路徑：
- 註冊一個新帳號
- 登入
- 讀一個 chunk
- 回答
- 查看 progress
- 修改 settings

### 8.4 前端串接測試
- 前端能否正確拿到 token
- 前端能否正確顯示後端資料
- 前端與後端之間的錯誤處理是否合理

---

## 9. 你需要提前考慮的風險

### 9.1 資料庫相關
- 沒有資料庫
- 連接字符串錯誤
- 遷移失敗
- 欄位名稱不一致

### 9.2 認證相關
- token 過期
- 無權訪問
- 密碼驗證失敗
- 前端送的 header 不對

### 9.3 內容相關
- chunk 資料缺失
- examples 不完整
- quiz 選項不合理
- 同一個內容被重複生成

### 9.4 AI 生成相關
- AI API 失敗
- 生成過慢
- 成本過高
- 生成結果不符合需求

### 9.5 前後端整合相關
- CORS 問題
- API 路徑不一致
- JSON 格式不一致
- 前端渲染和後端字段不匹配

---

## 10. 建議的實際開發順序

我建議你按這個順序做，最穩妥：

1. 先把資料庫和 auth 做好
2. 再把 chunk / quiz / example 相關接口做好
3. 再做 progress / settings
4. 再接 AI generation pipeline
5. 最後做前端整合

這樣每一步都是一個可驗證的里程碑。

---

## 11. 最終完成後的狀態

完成後，你的後端應該具備這些能力：

- 用戶可以註冊登入
- 用戶可以學習 chunk
- 系統會記錄他們的學習進度
- 系統會提供 quiz 練習
- 內容可以由 AI 批量生成並存入池子
- 前端可以直接消費這些 API

---

如果你要真正開始，我建議下一步先做這三件事：

1. 把 Prisma + PostgreSQL 接好
2. 完成 auth 真實登入註冊
3. 把 chunks 的資料從 mock 改成資料庫

這三步會是整個後端最關鍵的基礎。
